\section{SQL}

SQL è il linguaggio di riferimento per le basi di dati relazionali. Originariamente SQL
era l'acronimo di "Structured Query Language", ma oggi è semplicemente SQL. 
SQL ha varie funzionalità:
\begin{itemize}
    \item \textbf{Data Definition Language (DDL)}: consente di definire la struttura del database, 
    come la creazione di tabelle, l'aggiunta di colonne e la definizione di chiavi primarie.
    \item \textbf{Data Manipulation Language (DML)}: consente di manipolare i dati 
    all'interno del database, come l'inserimento, l'aggiornamento e la cancellazione di record.
    \item \textbf{Data Query Language (DQL)}: consente di interrogare il database 
    per recuperare informazioni specifiche.
\end{itemize}
Ne esistono inoltre varie versioni. In laboratorio useremo 
SQL-2 poiché sappiamo che è supportato da tutti i DBMS.

\subsection{Istruzioni}

La sintassi di base è:
\begin{verbatim}
    SELECT [ DISTINCT ]
    [ * | expression [[ AS] output_name ] [, ...] ]
    [ FROM from_item [, ...] ]
    [ WHERE condition ]
    [ GROUP BY grouping_element [, ...] ]
    [ HAVING condition [, ...] ]
    [ { UNION | INTERSECT | EXCEPT } [ DISTINCT ] other_select ]
    [ ORDER BY expression [ ASC | DESC | USING operator ]]
\end{verbatim}
\noindent
\begin{itemize}
    \item expression è un' espressione che determina un attributo.
    \item from$\_$item è un' espressione che determina una sorgente per gli attributi.
    \item condition è un' espressione booleana per selezionare i valori degli attributi.
    \item grouping$\_$element è un’espressione per poter eseguire operazioni su più valori di un attributo e
    considerare il risultato
    \item SELECT, FROM e WHERE sono chiamate clausole.
    \item Estrae dal prodotto cartesiano delle tabelle elencate nella clausola FROM, le righe che soddisfano la
    condizione specificata nella clausola WHERE.
    \begin{itemize}
        \item Il risultato dell‘interrogazione è una tabella con una riga per ogni riga prodotta dalla clausola FROM ed
        eventualmente filtrata dalla clausola WHERE (se esiste), le cui colonne dipendono dalla lista degli attributi
        nella clausola SELECT.
    \end{itemize}
\end{itemize}

\subsection{SELECT}

\begin{verbatim}
    SELECT [ DISTINCT ]
    [{* | expression [[ AS] output_name ]} [, ...]]
\end{verbatim}


\begin{itemize}
    \item è un’abbreviazione per indicare tutti gli attributi delle tabelle.
    \item expression è un’espressione che coinvolge gli attributi della tabella (può anche essere semplicemente
    il nome di un attributo).
    \item output$\_$name è il nome assegnato all’attributo che conterrà il risultato della valutazione
    dell’espressione expression nella relazione risultato.
    \item DISTINCT: se presente richiede l’eliminazione delle tuple duplicate.
\end{itemize}

\subsection{FROM}

\begin{verbatim}
    FROM from_item [, ...]
\end{verbatim}
\noindent

\begin{enumerate}
    \item \texttt{table\_name [[AS] alias [(column\_alias [, ...])]}
    \begin{itemize}
        \item Se sono presenti due o più tabelle, si esegue il prodotto cartesiano tra
         tutte le tabelle. Se ci sono attributi con lo stesso nome su tabelle
        diverse, nella SELECT questi attributi sono identificati anteponendo il nome
        della tabella e un ’.’: nomeTabella.nomeAttributo
    \end{itemize}
    \item \texttt{(other\_select) [AS] nomeRisultato [(column\_alias [,...])]}
    \begin{itemize}
        \item è una SELECT innestata. Il risultato della SELECT interna è lo schema con nome nomeRisultato su cui fare la SELECT corrente.
    \end{itemize}
    \item \texttt{from\_item [NATURAL] join\_type from\_item [ON condition]}
    \begin{itemize}
        \item è la clausola JOIN.
        \item Metodo più sofisticato che permette di selezionare 
        un sottoinsieme del prodotto cartesiano di due o più tabelle. join\_type specifica
        il tipo di join. I tipi principali sono riassunti nelle slide seguenti.
    \end{itemize}
\end{enumerate}

\subsection{WHERE}

\begin{verbatim}
    WHERE condition
\end{verbatim}
\noindent
La clausola WHERE ammette come argomento un’espressione booleana costruita
    combinando predicati semplici con gli operatori AND, OR e NOT
\begin{itemize}
    \item AND: seleziona le righe in cui tutti i predicati sono veri
    \item OR: seleziona le righe in cui almeno un predicato è vero
    \item NOT: seleziona le righe in cui il predicato è falso
\end{itemize}
\noindent
Ciascun predicato semplice usa gli operatori $= , <>, <, >, <=$ e $>=$ p
er confrontare un’espressione costruita a 
partire dal valore di un attributo con un valore costante,
oppure un’altra espressione.
Per esprimere la precedenza degli operatori AND e OR si fa uso della parentesi.

\subsubsection{LIKE}

Per il confronto tra stringhe, SQL mette a disposizione anche
l’operatore LIKE che consente verificare se una stringa rispetta un certo
“pattern” (oppure ILIKE per case insentive). \\
Due caratteri speciali per il confronto “$\_$” e “$\%$”:
\begin{itemize}
    \item “$\_$”: confronto con un carattere arbitrario
    \item “$\%$”: confronto con una stringa di lunghezza arbitraria (anche 0) di caratteri
    arbitrari
\end{itemize}
‘$\_o\%i$’ = stringa che ha una o in seconda posizione, seguita da una
sequenza qualsiasi di caratteri (anche vuota) e che termina con una i.

\subsubsection{BETWEEN}

L’operatore BETWEEN consente di verificare se un valore è compreso tra due valori
\begin{verbatim}
    WHERE attributo [NOT] BETWEEN valore1 AND valore2;
\end{verbatim}
\noindent
Il predicato ritorna true se il valore di attributo è compreso tra valore1 e
valore2 (inclusi). Se è presente NOT, ritorna true se il valore di attributo è
minore di valore1 o maggiore di valore2.

\subsubsection{SIMILAR TO}

L’operatore SIMILAR TO è un LIKE più espressivo che accetta,
come pattern, un sottoinsieme delle espressioni regolari POSIX
(versione SQL). \\
Esempi di componenti di espressioni regolari:
\begin{itemize}
    \item $\_$: confronto con un carattere arbitrario
    \item $\%$: 0 o più caratteri arbitrari
    \item $\*$: ripetizione dell’elemento precedente (carattere o gruppo) 0 o più volte
    \item $+$: ripetizione dell’elemento precedente (carattere o gruppo) 1 o più volte
    \item $\{n,m\}$: ripetizione del precedente match almeno n e non più di m volte. 
    \item $[...]$: elenco di caratteri ammissibili
\end{itemize}

\subsubsection{IN}

Nella clausola WHERE può apparire l’operatore [NOT] IN per
testare l’appartenenza di un valore ad un insieme.

\begin{verbatim}
    WHERE attributo [NOT] IN ( valore [, ...] );
\end{verbatim}

\subsubsection{IS NULL}

Per verificare se un attributo ha un valore nullo o meno si utilizza il predicato
IS NULL.

\begin{itemize}
    \item Attributo IS NULL: ritorna true se l’attributo ha valore nullo
    \item IS NOT NULL: è la negazione, ritorna true se l’attributo ha valore diverso da
    nullo
\end{itemize}

\begin{verbatim}
    WHERE attributo IS [ NOT ] NULL;
\end{verbatim}

SQL-2 implementa la logica a 3 valori:
\begin{itemize}
    \item Un predicato semplice restituisce il valore NULL quando uno qualsiasi dei termini del
    predicato ha valore nullo.
    \item Il predicato IS NULL è una eccezione, restituendo sempre il valore vero oppure false, mai
    il valore NULL.
\end{itemize}

\subsection{SQL e Algebra Relazionale}

\subsubsection{Duplicati}

Una significativa differenza tra SQL e algebra relazionale è la gestione dei duplicati.
L’operazioni di eliminazione delle righe doppie 
(ad esempio al seguito di una selezione con where) è molto costosa, quindi di default SQL
mantiene le righe duplicate.

\begin{itemize}
    \item Il costrutto DISTINCT dopo la parola chiave SELECT permette di
    eliminare i duplicati.
    \item Il costrutto ALL permette invece di mantenerli, ma questa è già la scelta di default.
\end{itemize}

\subsubsection{Selezione, Proiezione, Join}

\begin{itemize}
    \item L’istruzione SELECT, con una sola relazione nella clausola FROM
    permette di realizzare: selezioni, proiezioni, ridenominazioni.
    \item con più relazioni nella clausola FROM si realizzano join (e prodotti
cartesiani)
\end{itemize}
\noindent
In SQL il join si può definire in due modi diversi:
\begin{itemize}
    \item Il primo non richiede l’uso di nuova sintassi, e specifica il legame
    tra le tabelle coinvolte a livello della clausola WHERE.
    \item Il secondo usa una sintassi riservata a livello di clausola FROM
    (useremo questo).
\end{itemize}
La clausola FROM ammette come argomento: \texttt{table\_name [[AS] name]] [, ...]}.
Si è visto che se sono presenti due o più nomi di tabelle, si esegue
il prodotto cartesiano tra tutte le tabelle e lo schema del risultato
può contenere tutti gli attributi del prodotto cartesiano.
Il prodotto cartesiano di due o più tabelle è un \textbf{CROSS JOIN}.
A partire da SQL-2, esistono altri tipi di JOIN (join\_type):
INNER JOIN, LEFT [OUTER] JOIN, RIGHT [OUTER] JOIN e FULL
[OUTER] JOIN.

\subsection{JOIN}

\begin{verbatim}
    table_name [ NATURAL ] join_type table_name [ON join_condition [, ...]].
\end{verbatim}
\noindent
dove join\_condition è un’espressione booleana che
seleziona le tuple del join da aggiungere al risultato. Le
tuple selezionate possono essere poi filtrate con la
condizione della clausola WHERE.
Il join si può spostare nella clausola from, a livello delle tabelle (nello
svolgere gli esercizi, useremo questa sintassi dedicata).

\begin{itemize}
    \item \begin{verbatim}
        SELECT ListaAttributi
        FROM Tabella1 JOIN Tabella2 ON ( CondizioneDiJoin )
        [ WHERE AltraCondizione]
    \end{verbatim}
    \item \begin{verbatim}
        SELECT i.Nome, d.Città
        FROM Impiegato AS i JOIN Dipartimento as d ON ( i.Dipart = d.Nome )
    \end{verbatim}
\end{itemize}
\noindent
Le parole OUTER, LEFT, RIGHT e FULL sono opzionali. Se non sono presenti, si intende INNER JOIN.

\subsubsection{INNER JOIN}

\begin{verbatim}
    table1 [ INNER ] JOIN table2 ON join_condition [, ...].
\end{verbatim}
\noindent
Rappresenta il tradizionale $\theta$-join dell’algebra relazionale.
Combina ciascuna riga r1 di table1 con ciascuna riga di
table2 che soddisfa la condizione della clausola ON.

\subsubsection{LEFT [OUTER] JOIN}

\begin{verbatim}
    table1 LEFT [ OUTER ] JOIN table2 ON join_condition [, ...].
\end{verbatim}
\noindent
Si esegue un INNER JOIN. Poi, per ciascuna riga r1 di
table1 che non soddisfa la condizione con qualsiasi
riga di table2, si aggiunge una riga al risultato con i
valori di r1 e assegnando NULL agli altri attributi...

\begin{Warning}{Attenzione}{}
Il LEFT [OUTER] JOIN non è simmetrico! Con le medisime tabelle si possono ottenere risultati diversi invertendo l’ordine delle
tabelle nel join! 
\end{Warning}

\subsubsection{RIGHT [OUTER] JOIN}

\begin{verbatim}
    table1 RIGHT [ OUTER ] JOIN table2 ON join_condition [, ...].
\end{verbatim}
\noindent
Si esegue un INNER JOIN. Poi, per ciascuna riga r2 di
table2 che non soddisfa la condizione con qualsiasi
riga di table1, si aggiunge una riga al risultato con i
valori di r2 e assegnando NULL agli altri attributi..

\subsubsection{FULL [OUTER] JOIN}

\begin{verbatim}
    table1 FULL [ OUTER ] JOIN table2 ON join_condition [, ...].
\end{verbatim}

È equivalente a: INNER JOIN + LEFT OUTER JOIN + RIGHT OUTER JOIN.

\subsection{ALIAS}

SQL permette di associare un nome alternativo (alias) alle tabelle
che compaiono nella clausola FROM. Il nome viene usato per far riferimento alla tabella nel contesto
dell’interrogazione. Questo perché:
\begin{itemize}
    \item Far riferimento alla tabella in modo compatto.
    \item Fare accesso più volte alla stessa tabella
\end{itemize}

\subsection{ORDER BY}

Come abbiamo visto, una relazione è un insieme di tuple non ordinato. Nella
realtà, però, sorge spesso il bisogno di costruire un ordine sulle righe.
SQL permette di specificare un ordinamento sulle righe dell'output grazie alla
clausola ORDER BY.
\begin{verbatim}
    ORDER BY attributo [{ ASC | DESC }] [, ... ];
\end{verbatim}
\noindent
Le righe vengono ordinate per il primo attributo nell’elenco, a parità di valore
del primo attributo si usa il secondo, e così via in sequenza.

\subsection{OPERAZIONI INSIEMISTICHE}

SQL mette a disposizione degli operatori insiemistici. 
\begin{verbatim}
    SELECT
    { UNION | INTERSECT| EXCEPT
    [ALL] SELECT }
\end{verbatim}
Gli operatori insiemistici, al contrario del resto del linguaggio,
assumono come default di eseguire un’eliminazione dei duplicati, a
meno che si usi ALL.\\
Sono sufficienti domini compatibili ed ugual numero di attributi
\begin{itemize}
    \item Ordine posizionale: la corrispondenza degli attributi si basa sulla posizione
    \item Se i nomi degli attributi sono diversi di solito il sistema usa il nome del primo
    operando
\end{itemize}

\subsection{OPERATORI AGGREGATI}

Una delle più importanti estensioni di SQL rispetto all’algebra
relazionale: \textbf{COUNT, SUM, AVG, MIN, MAX}.
In algebra relazionale tutte le condizioni specificate vengono
valutate su una tupla alla volta.
Gli operatori di aggregazione permettono di valutare proprietà che
dipendono da insiemi di tuple. Prima viene eseguita l’interrogazione considerando 
solo le clausole FROM e WHERE, quindi l’operatore di aggregazione viene applicato alla
tabella contenente il risultato.

\subsubsection{COUNT}

L’operatore COUNT conta il numero di righe di una tabella o il
numero di valori di uno o più attributi:

\begin{verbatim}
    COUNT ( <* | [DISTNCT | ALL] ListaAttributi> )
\end{verbatim}
\noindent
\begin{itemize}
    \item *: conta il numero di righe della tabella
    \item DISTINCT ListaAttributi: conta il numero di valori diversi degli attributi ListaAttributi
    \item ALL ListaAttributi: conta il numero di righe che hanno valori diversi
    da NULL negli attributi in ListaAttributi (comportamento di default)
\end{itemize}

\subsubsection{MAX, MIN, AVG, SUM}

\begin{verbatim}
    <SUM | MAX | MIN | AVG > ( [DISTINCT | ALL] AttrExpr )
\end{verbatim}

\begin{itemize}
    \item SUM: calcola la somma dei valori di AttrExpr
    \item MAX: calcola il valore massimo di AttrExpr
    \item MIN: calcola il valore minimo di AttrExpr
    \item AVG: calcola la media dei valori di AttrExpr
\end{itemize}
\noindent
\textbf{SUM} e \textbf{AVG} ammettono come argomento solo espressioni che
rappresentano valori numerici o intervalli di tempo.
\textbf{MAX} e \textbf{MIN} ammettono come argomento solo espressioni su cui è
definito un ordinamento (quindi anche stringhe). \textbf{DISTINCT} 
elimina i duplicati mentre \textbf{ALL} trascura solo i valori NULL. (DISTINCT e ALL non hanno effetto su MIN e MAX)
\begin{verbatim}
    SELECT Cognone, Nome, MAX( I.Stipendio )
    FROM Impiegato as I JOIN Diparmento as D
    ON I.Dipart = D.Nome
    WHERE D.Città = ‘Milano’
\end{verbatim}
\begin{Warning}{}{}
la precedente query non è corretta!
Gli operatori aggregati non sono un meccanismo di selezione, ma solo
funzioni che restituiscono un valore quando sono applicate ad un insieme.
Cognome e Nome restituiscono un valore per ogni tupla selezionata, mentre
MAX() restituisce un valore unico per tutto l’insieme. Per risolvere questa situazione serve il meccanismo di raggruppamento che
non vediamo.
\end{Warning}

\subsection{OPERATORI DI RAGGRUPPAMENTO}

\subsubsection{GROUP BY}

Gli operatori aggregate possono essere applicati separatamente a
sottoinsiemi di righe.
La clausola GROUP BY permette di specificare come dividere le
tabelle in sottoinsiemi. La clausola permette di specificare un insieme di attributi e
l’interrogazione raggrupperà le righe che possiedono gli stessi
valori per questo insieme di attributi.
In una interrogazione che fa uso della clausola GROUP BY può comparire
come argomento della SELECT solo un sottoinsieme degli attributi usati nella
clausola GROUP BY oppure operatori di aggregazione.

\subsubsection{HAVING}

È possibile applicare delle selezioni ai gruppi generati tramite la
clausola GROUP BY. La clausola HAVING permette di specificare delle condizioni che
devono essere soddisfatte dai gruppi generati dopo l’esecuzione
del raggruppamento tramite GROUP BY.

\begin{Observation}{}{}
    \begin{itemize}
        \item La clausola WHERE applica delle selezioni su SINGOLE RIGHE, prima
        dell’eventuale raggruppamento tramite GROUP BY.
        \item La clausola HAVING applica delle selezioni su GRUPPI, generati a
        seguito dell’esecuzione del raggruppamento tramite GROUP BY.
    \end{itemize}

    \begin{enumerate}
        \item WHERE $\to$ seleziona le righe che
        verranno raggruppate
        \item GROUP BY $\to$ raggruppa le righe selezionate
        \item HAVING $\to$ seleziona i gruppi che fanno parte del risultato
    \end{enumerate}
\end{Observation}

\subsection{Interrogazioni nidificate}

Una interrogazione nidificata è una interrogazione che contiene al
suo interno altre interrogazioni.
La nidificazione può avvenire sia nella clausola SELECT, che nella
clausola FROM, che nella clausola WHERE.
Il caso tipico è la nidificazione nella clausola WHERE, dove un
predicato contiene un confronto tra un valore (espressione su una
singola riga) ed il risultato di un’interrogazione SQL.

\subsubsection{Nidificazione nella clausola WHERE}

Se in un predicato si confronta un attributo o una espressione che
produce un valore singolo con il risultato di un’interrogazione,
sorge il problema di disomogeneità dei termini del confronto.
I normali operatori di confronto ($=, <>, <, >, <=, =>$) possono
essere estesi con le parole chiave ANY e ALL.
\begin{itemize}
    \item ANY: la riga soddisfa la condizione se risulta vero il confronto (con
    l’operatore specificato) tra il valore dell’attributo (o espressione) e
    \textbf{ALMENO} uno degli elementi restituiti dall’interrogazione.
    \item ALL: la riga soddisfa la condizione se risulta vero il confronto (con
    l’operatore specificato) tra il valore dell’attributo (o espression) e
    \textbf{TUTTI} gli elementi restituiti dall’interrogazione
\end{itemize}
\noindent
Per le interrogazioni nidificate viste fino ad ora, l’interrogazione
può essere eseguita una volta sola ed il risultato ottenuto può
essere salvato in una tabella temporanea usata per eseguire
ciascun confronto con le righe dalla tabella esterna.
Talvolta però l’interrogazione nidificata fa riferimento al contesto
dell’interrogazione che la racchiude attraverso l’uso di una
variabile (\textit{passaggio di binding}).
L’interrogazione nidificata dovrà essere valutata separatamente
per ciascuna riga prodotta dalla valutazione della query esterna.

\subsubsection{Nidificazione nidifiicate Binding e EXISTS}

L’operatore logico \textbf{EXISTS} riceve come parametro
un’interrogazione nidificata e restituisce il valore vero solo se
l’interrogazione fornisce un risultato non vuoto.
Questo operatore può essere usato in modo significativo solo
quando si ha un passaggio di binding tra la query esterna e quella
interna.

\begin{verbatim}
SELECT * FROM Persona P
WHERE EXISTS (SELECT * FROM Persona P1 WHERE P1.Nome 
    = P.Nome AND P1.Cognome = P.Cognome AND P1.CodFiscale 
    <> P.CodFiscale );
\end{verbatim}
Nel codice sopra si estraggono le persone che hanno degli omonimi (ovvero
persone con lo stesso nome e cognome, ma codice fiscale
diverso).
In questo prossimo codice si vanno ad estrarre le persone che non hanno degli omonimi (ovvero
persone con lo stesso nome e cognome, ma codice fiscale
diverso)

\begin{verbatim}
SELECT * FROM Persona P
WHERE NOT EXISTS (SELECT * FROM Persona P1
    WHERE P1.Nome = P.Nome AND
    P1.Cognome = P.Cognome AND
    P1.CodFiscale <> P.CodFiscale );
\end{verbatim}
\noindent
Le interrogazioni nidificate possono essere specificate anche come
elemento della clausola SELECT.
Spesso l’uso di query nidificate nella SELECT sostituiscono l’uso del JOIN.
Per esempio, estrarre per ogni cantante il numero di canzoni di cui è autore.
\begin{verbatim}
SELECT DISTINCT Nome, NumCanzoni =
    (SELECT COUNT(*) FROM Autore A
    WHERE A.Nome = C.Nome)
    FROM Cantante C
\end{verbatim}
\noindent
oppure
\begin{verbatim}
SELECT DISTINCT C.Nome, COUNT(*) as NumCanzoni
FROM Cantante C JOIN Autore A
ON C.Nome = A.Nome
GROUP BY C.Nome
\end{verbatim}

\begin{Observation}{}{}
Ricordiamo che è necessario che l’interrogazione nidificate restituisca una
sola tupla (riferimenti tramite variabile alla interrogazione esterna).
\end{Observation}
\noindent
Le interrogazioni nidificate possono essere utilizzate nella
clausola FROM come sorgente dati su cui eseguire
l’interrogazione. È possibile creare una catena di interrogazioni, dove ciascuna
interrogazione utilizza il risultato della precedente come una
tabella di base. 
Per esempio: si estrae per ogni cantante che è anche autore di canzoni il numero
di canzoni di cui è rispettivamente interprete e autore.
\begin{verbatim}
SELECT DISTINCT C.Nome, C.NumCantate, A.NumScritte
FROM
(SELECT Nome, Count(*) as NumCantate FROM Cantante) C
JOIN
(SELECT Nome, Count(*) as NumScritte FROM Autore ) A
ON C.Nome = A.Nome
\end{verbatim}
\noindent
Prima vengono eseguite le interrogazioni nella clausola FROM e i
risultati di queste vengono usati come normali tabelle.

\subsection{VIEW}

Le viste permettono di salvare una query che genera una tabella temporanea.

\begin{lstlisting}[language=SQL]
CREATE VIEW <nome_vista> AS (<QUERY>)
\end{lstlisting}
\noindent
Dato un nome alla query, la possiamo usare come se fosse il nome di una tabella.

\subsection{Esercizi}

\begin{Example}{}{}
We have here three tables:
\begin{itemize}
    \item \textbf{TURISTA} con gli attributi USERNAME, COGNOME, NOME, DATA\_NASCITA, NAZIONALITA
    \item \textbf{ATTRAZIONE} con gli attributi CODICE, NOME, DESCRIZIONE, LATITUDINE, LONGITUDINE, INDIRIZZO, CITTA, PROV
    \item \textbf{RECENSIONE} con gli attributi ATTRAZIONE, TURISTA, TIMESTAMP, TITOLO, TESTO, VOTO
\end{itemize}
\begin{enumerate}
    \item Trovare tutte le attrazioni della città di Verona il cui nome inizia con la lettera
    $f$ maiuscola o minuscola. 
\begin{lstlisting}[language=SQL]
SELECT *
FROM ATTRAZIONE
WHERE CITTA = 'Verona' AND NOME ILIKE 'f%';
\end{lstlisting}
    \item Trovare tutti i turisti che nel 2025 non hanno mai rilasciato
    recensioni con un voto $>$ 2 per attrazioni fuori dalla provincia di Verona.
\begin{lstlisting}[language=SQL]
SELECT T.USERNAME FROM TURIST T 
EXCEPT (

\subsection{Operazioni sulle pagine} 

\begin{itemize}
    \item \textbf{Inserimento}
    \begin{itemize}
        \item Se esiste uno spazio contiguo sufficiente $\Rightarrow$ siamo nel caso semplice 
        perché ci basta aggiungere una tupla e aggiornare il dizionario. 
        \item Se non esiste uno spazio sufficiente $\Rightarrow$ 
        dobbiamo avere un'interazione con il filesyste per allocare nuovi blocchi per
        i file. 
        \item Se esiste uno spazio sufficiente ma non contiguo $\Rightarrow$
        Riorganizzazione spazio: inserimento semplice.
    \end{itemize}
    \item \textbf{Cancellazione}
    \begin{itemize}
        \item Sempre possibile e non necessita riorganizzazione
    \end{itemize}
    \item  \textbf{Accesso}
    \begin{itemize}
        \item Tupla $\leadsto$ dizionario, accesso sequenziale  
        \item Ad un attributo di una tupla: 
        \begin{itemize}
            \item Identificare la tupla $t$ 
            \item Scorrere elementi di $t$ tramite offset
        \end{itemize}
    \end{itemize}
\end{itemize}

\subsection{Criteri di organizzazione}

I criteri per cui organizziamo le tuple nei blocchi possono essere:

\begin{itemize}
    \item \textbf{Sequenziale}: abbiamo una disposizione consecutiva 
    delle tuple in base ad un criterio specifico (es. ordine di inserimento). 
    \item \textbf{Accesso calcolato (Hash)}: La posizione delle tuple nei blocchi
    dipende da un algoritmo di hashing. 
    \item \textbf{Albero}: principato utilizzato per la costruzione di indici. 
\end{itemize}

\subsubsection{Struttura primaria sequenziale/seriale}

Nella struttura primaria sequenziale, costruiamo i blocchi 
"logicamente" consecutivi. Assumiamo che questi blocchi siano uno vicini all'altro,
in modo da minimizzare i tempi di accesso.
Ci sono due modi per organizzare le tuple nei blocchi:
\begin{itemize}
    \item \textbf{Disordinata}: le tuple sono disposte rispetto all'ordine di inserimento.
    È la forma più semplice con cui salviamo i nostri dati. 
    Quali sono le sue caratteristiche?
    \begin{itemize}
        \item \textbf{Inserimento:} prendo l'ultimo blocco ($b$) del file libero e tento di fare 
        l'inserimento $b$. Ha bisogno di un solo accesso a memoria secondaria. 
        \item \textbf{Ricerca:} non è efficiente perché dovremmo fare una scansione 
        sequenziale di tutti i blocchi del file. 
        \item \textbf{Cancellazione/Modifica:} Per potere eliminare una tupla 
        abbiamo prima bisogno di trovarla. Dunque richiede la ricerca. Una volta
        che l'abbiamo trovata, basta semplicemente che cancelliamo senza ricompattare. 
        Nella modifica invece può cambiare la dimensione della tepla e potrebbe 
        non starci nello stesso spazio. In questo caso,
        devo cancellare e inserire. Per far fronte a questo problema,
        il DBMS può riorganizzare i blocchi periodicamente. 
    \end{itemize}
    \item \textbf{Ordinata}: le tuple sono disposte rispetto ad un campo chiamato "chiave di ordinamento".
    Questa struttura rende efficiente le operazioni che hanno bisogno dell'ordinamento 
    utilizzato ad esempio:
    \begin{itemize}
        \item Elenco ordinato
        \item Range query 
        \item Operazioni aggregate 
    \end{itemize}
    In caso di aggiornammento è necessario mantenere l'ordinamento,
    quindi servono periodiche riorganizzazioni dei blocchi.
    \begin{itemize}
        \item \textbf{Inserimento: } 
        \begin{itemize}
            \item Quando devo fare l'inserimento
            devo prima individuare il blocco in cui fare l'inserimento. 
            Devo andare a pescare il blocco candidato ovvero quello che contiene
            la tupla precedente nell'ordinamento.
            \item Se c'è spazio, allora faccio l'inserimento e basta, altrimenti
            devo aggiungere un nuovo blocco in cui mettere questa tupla.
        \end{itemize}
        \item \textbf{Cancellazione: }
        \begin{itemize}
            \item Devo individuare il blocco $b$ che la contiene
            \item Cancellare la tupla 
            \item Aggiusto i puntatori tra i blocchi per mantenere l'ordinamento
        \end{itemize}
        \item \textbf{Riorganizzazione: }
        TODO 
    \end{itemize}
\end{itemize}

\subsection{Indici}

Gli indici sono delle strutture dati per l'accesso efficiente ai file sequenziali. 
L'obiettivo degli indici è quello di velocizzare l'accesso casuale 
via chiave di ricerca (un insieme di attributi).
Gli indici si contraddistinguono tra due tipi:
\begin{itemize}
    \item \textbf{Indici primari}: la chiave di ordinamento è uguale alla chiave di ricerca. 
    Ogni record dell'indice è composto da:
    \[
        < v_i, p_i >
    \]
    dove $v_i$ è il valore della chiave di ricerca e $p_i$ è il puntatore 
    alla prima tupla con quel valore di chiave (basta 1 puntatore perché chiave 
    ricerca = chiave ordinamento). Trovata la prima tupla, le altre sono di seguito.
    Inoltre l'indice può essere 
    \begin{itemize}
        \item \textbf{denso}, ovvero che esiste un recordo di indice per ogni chiave di ricerca.
        \item \textbf{sparso}, solo per alcune occorrenze dlle chiave persenti nel file esiste un corrispondente 
        record nell'indice, tipicamente una per blocco.
    \end{itemize}
    \begin{itemize}
        \item Per la \textbf{ricerca} di una chiave $k$ se è \textbf{denso}:
        \begin{itemize}
            \item $K$ è presente nell'indice?
            Se no, allora è finito subito e ho solo un acccesso all'indice (vuol dire 
            fare una scansione sequenziale dei blocchi in cui è salvato l'indice).
            Altrimenti, se è presente, faccio accesso al blocco puntato da
            $p_k$ e quindi il mio costo equivale a un accesso all'indice + 
            un accesso al blocco dati. Per l'accesso all'indice devo comunque 
            caricar el'indice di memoria che sono comunque molto meno rispetto
            a fare una scansione sequenziale. 
        \end{itemize}
        \item Per la \textbf{ricerca} di una chiave $K$ se è \textbf{sparso} ($k$ potrebbe 
        non essere presente nell'indice ma essere presente nelle tuple. ):
        \begin{itemize}
            \item Eseguo una scansione sequenziale dell'indice per trovare $(k', p_k)$
            dove $k'$ è il valore di chiave più grande minore 
            o uguale alla chiave $k$ che sto cercando
            \item A questo punto seguo il puntatore $p_k$ per accedere al blocco.
            In questo caso il costo dell'operazione è:
            \[
                \text{costo = un accesso all'indice + 
            un accesso al blocco dei dati. }
            \]
        \end{itemize}
        \item \textbf{Inserimento}:
        \begin{itemize}
            \item Come un inserimento su file sequenziale disordinato/ordinato
            \item Aggiornare indice (mantenere l'indice è costoso, quindi 
            bisogna fare dei buon compromessi su il numero di indici che costruiamo
            e il numero di attributi su cui non calcoliamo l'indice poiché 
            non ha senso)
            \begin{itemize}
                \item Se è denso, inserire una coppia nel file indice solo se la chiave 
                è nuova 
                \item Se è sparso, inserire una coppia solo se l'inserimento della 
                tupla aggiunge un nuovo blocco 
            \end{itemize}
        \end{itemize}
        \item \textbf{Cancellazione}: 
        \begin{itemize}
            \item Cancellazione dal file sequenziale disordinato/ordinato 
            \item Aggiorniamo l'indice:
            \begin{itemize}
                \item Se è denso, cancelliamo anche la coppia 
                nel file di indice solo se la tupla era l'ultima con chiave $k$.  
                \item Se è sparso procediamo a cancellare anche la coppia nel file 
                di indice se la tupla era l'ultima con chiave $k$ 
                inoltre aggiungere una nuova coppia con chiave $k'$ (chiave 
                più grande minore o uguale a $k$) se $k'$ è diverso da $k$.
            \end{itemize}
        \end{itemize}
    \end{itemize}
    \item \textbf{Indici secondari}: la chiave di ordinamento è diversa 
    dalla chiave di ricerca. 
    Ogni record dell'indice è composto da:
    \[
        < v_i, p_i >
    \]
dove $v_i$ è il valore della chiave di ricerca e $p_i$ è un puntatore ad un bucket
di puntatori che individuano nel file sequenziale tutte le tuple con chiave $v_i$. 
Gli indice secondari \textbf{sono sempre densi} perché non si può sfruttare 
l'ordinamento delle tuple. \\
Le operazioni che si possono eseguire su un indice secondario sono:
\begin{itemize}
    \item \textbf{Ricerca} di una chiave $k$:
    \begin{itemize}
        \item Ricerca sequenziale per trovare $k \Rightarrow (k, p_k)$:
        \begin{itemize}
            \item Se è denso, se $k$ non c'è, non esistono tuple con chiave $k$ 
        \end{itemize}
        \item Accesso al bucket $b$ $p_k$
        \item Accesso al file tramite i puntatori in $b$.
    \end{itemize}
    Il costo dell'operazione è:
    \[
        \text{costo = un accesso all'indice + un accesso al bucket + $n$ accessi al file}
    \]
    dove $n$ è uguale al numero di puntatori in $b$. 
    \end{itemize}
    \item \textbf{Inserimento/Cancellazione}: rimane come un indice primario 
    denso ma con aggiornamento dei bucket. 
\end{itemize}

\subsection{Minimizzare i costi di accesso}

Per migliorare l'accesso agli indici, possiamo migliorare le prestazioni 
utilizzando la struttura ad albero (B+TREE) oppure le strutture ad accesso 
calcolato (Hash).

\subsubsection{B+TREE}

\begin{itemize}
    \item Struttura ad albero 
    \item Ogni nodo viene memorizzato in una pagina/blocco di memoria 
    secondaria.
    \item Ogni nodo ci sono dei legami che rappresentano i puntatori alle pagine. 
    \item È caratterizzato per avere pochi livelli e molti nodi foglia. 
    \item È un albero bilanciato in modo che non esistono dei path preferenziali o 
    più "fortunati" rispetto agli altri. Inserimenti e cancellazioni non devono alterare 
    le prestazioni di accesso ai dati. 
\end{itemize}

\subsubsection*{Struttura dell'albero}

\begin{itemize}
    \item \textbf{Nodi foglia: } (con fan-out = $n$) Può contenere fino a $n - 1$ 
    valori ordinati di chiave di ricerca e fino ad $n$ puntatori ai dati. 

    \begin{figure}[H]
        \centering 
        \includegraphics[width=0.7\textwidth]{btree_f.png}
        \caption{Struttura di un B+TREE con fan-out = n}
    \end{figure}
    \noindent
    Ogni nodo foglia ha un \textit{vincolo di ordinamento}. 
    Questo vuol dire che le chiave dentro ad un nodo foglia sono ordinate 
    e inoltre sono ordinate rispetto alle chiavi degli altri nodi foglia.
    \[
        \forall L_i, L_j \text{ nodi foglia } i < j: 
        \forall k_t \in L_i, k_j \in L_j: k_i < k_j
    \]
    Il puntatore $p_m$ del nodo $L_i$ punta al nodo $L_{i+1}$, ovvero al nodo foglia successivo
    se esiste e altrimenti è nullo.

    \begin{figure}[H]
        \centering 
        \includegraphics[width=0.7\textwidth]{btree_pm.png}
        \caption{Puntatore $p_m$ del nodo $L_i$ che punta al nodo $L_{i+1}$}
    \end{figure}

    \item \textbf{Nodi intermedi} (con fan-out = $n$)
    è una sequenza di $m' \le n$ valori ordinati di chiave. Può contenere fino a $n$ puntatori a nodo.
    Ogni chiave $K_i$ è seguita da un puntatore $p_i$: 

    \begin{figure}[H]
        \centering 
        \includegraphics[width=0.7\textwidth]{btree_int.png}
        \caption{Struttura di un nodo intermedio di un B+TREE con fan-out = n}
    \end{figure}
    \noindent
\end{itemize}
\noindent
Entrambi i due nodi devono rispettare il vincolo di reimpimento. 
\begin{itemize}
    \item Per un nodo foglia con fanout = $n$, ogni nodo foglia 
    può contenere un numero di valori di chiae (\#chiavi) vincolato come:
    \[
        \left\lceil \frac{n-1}{2} \right\rceil \le \colorbox{yellow}{\#\text{chiavi}} \le n - 1
    \]
    \item Per un nodo intermedio con fanout = $n$, ogni nodo intermedio
    contiene un numero di puntatori (\#puntatori) vincolato come:
    \[
        \left\lceil \frac{n}{2} \right\rceil \le \colorbox{green!90!black}{\#\text{puntatori}} \le n
    \]
\end{itemize}

\begin{Example}{}{}
 Esempio di B+Tree con contenuti (A, B, C, D, E, F, G, L, M, N, P) e
 con fanout = 4. Il fanout ci dice che ci sta un vincolo di reimpimento 
 dove i nodi foglia devono essere:
 \[
    2 \le \text{ \#chiavi} \le 3
 \]   
e i nodi intermedi devono essere:
\[
    2 \le \text{ \#puntatori} \le 4
\]

\begin{figure}[H]
    \centering 
    \includegraphics[width=1\textwidth]{btree_ex.png}
    \caption{Esempio di B+Tree con fanout = 4}
\end{figure}
\noindent
Questo esempio di albero per chiavi desiderate che rispetta
il vincolo di reimpimento usando un reimpimento \textbf{minimo}.
Tuttavia si poteva gestire il reimpimento in modo diverso, ad esempio con un reimpimento \textbf{massimo} 
che avrebbe portato ad avere nodi con 3 chiavi e 4 puntatori.

\begin{figure}[H]
    \centering 
    \includegraphics[width=1\textwidth]{btree_ex_2.png}
    \caption{Esempio di B+Tree con fanout = 4 e reimpimento massimo}
\end{figure}
\noindent
In questo caso non è necessario passare da un livello intermedio, possiamo rappresentare
tutto con unico nodo radice, perché puntare a $9$ nodi foglia ci servono 
$4$ puntatori e questo è compatibile con il vincolo di reimpimento.
\end{Example}

\begin{Observation}{Costo di una ricerca}
    Il costo di una ricerca di un valore $K$ in un B+TREE è dato da:
    \begin{itemize}
        \item La profondità dell'albero (ovvero il numero di nodi intermedi che dobbiamo attraversare)
        Questo dipende dal \textit{fanout}.
        La profondità è limitata da:
        \[
            prof_{B+Tree} \le \underbrace{1}_{\text{radice}} + \underbrace{\log_{\lceil \frac{n}{2} \rceil} \underbrace{\left( \frac{\text{\#valoriChiave}}{\lceil \frac{n-1}{2} \rceil} \right)}_{\text{MAX \# FOGLIE}}}_{\text{\# livelli}}
        \]
        \item \# di accessi alla memoria secondaria + ogni nodo è un blocco
    \end{itemize}
\end{Observation}

\subsubsection{Inserimento in un B+TREE}

\begin{enumerate}
    \item Ricerca del nodo foglia $NF$ dove $K$ va inserito. 
    \item Ci possono essere vari casi:
    \begin{itemize}
        \item Se $K$ è presente in $NF$:
        \begin{itemize}
            \item Se è un indice primario, allora nessuna azione. 
            \item Se è un indice secondario, allora aggiornare il bucket dei puntatori.
        \end{itemize}
        \item Se $K$ non è presente in $NF$, inserire $K$ in $NF$ rispettando ordine:
        \begin{itemize}
            \item Se è un indice primario, inserire $K$ e il puntatore alla tupla.
            \item Se è un indice secondario, inserire $K$ e creare un nuovo bucket
             che sarà puntato all'interno dell'indice.
        \end{itemize}
    \end{itemize}
    Se violo il vincolo di riempimento massimo, devo dividere $NF$ in due nodi foglia.
    \item Passo split: se in $NF$ esistono già $n$ valori di chiave:
    \begin{itemize}
        \item Creare due nodi foglia $NF_1$ e $NF_2$.
        \item Devo inserire i primi $\lceil \frac{n-1}{2} \rceil$ valori di chiave 
        in $NF_1$ e i restanti in $NF_2$.
        \item Aggiornare il padre. Se il padre viola il vincolo di riempimento massimo, 
        devo propagare lo split. Questa propagazione può arrivare fino alla radice 
        e in questo caso devo creare un nuovo nodo radice.
    \end{itemize}
\end{enumerate}

\subsubsection{Cancellazione}

\begin{enumerate}
    \item Ricerca del nodo foglia $NF$ che contiene $K$.
    \item Cancellare $K$ ed il suo puntatore:
    \begin{itemize}
        \item Potrebbe violare il vincolo di riempimento minimo. 
        Dopo la cancellazione:
        \[
            \text{ \#chiavi} < \left\lceil \frac{n-1}{2} \right\rceil
        \]
        In questo caso dobbiamo ricorrere all'operazione opposta dello split:
        il \textbf{merge}.
        Per effettuare il merge:
        \begin{itemize}
            \item Individuiamo il nodo fratello. 
            \item Si genera un unico nodo foglia. Se non esiste il fratello che unito 
            rispetta il vincolo di riempimento max, merge + split.
            \item Si aggiorna il nodo padre
            \item Se anche il padre viola il vincolo di riempimento minimo, merge anche per il padre.
        \end{itemize}
    \end{itemize}
\end{enumerate}

\subsubsection{Hash}

Questa struttura sfrutta l'accesso associativo nel senso 
che la localizzazione fisica dei dati diende dal valore della chiave. 
Viene utilizzata una funzione di hash che mappa i valori di chiave 
in indirizzi di memorizzazione delle tuple. 
\[
    h : K \to B 
\]
dove $K$ è il dominio delle chiavi e $B$ è il dominio degli indirizzi.
Costo della ricerca è uguale quindi ad un accesso alla memoria secondaria. 
Tuttavia il problema dell'hashing sono le collisioni: valori di chiavi diverse 
possono produrre lo stesso indirizzo.
Una funzione di hash è buona se minimizza il numero di collisioni. 
Viene definito il concetto di fattore di riempimento. 
\dfn{}
{
    \textbf{Il fattore di riempimento} mi dice lo spazio medio fisico utilizato 
    in ciascun blocco. 
}
\noindent
Consideriamo:
\begin{itemize}
    \item $T = \#\text{tuple}$ 
    \item $F = \#\text{tuple ci stanno in un blocco}$
    \item $f = \text{fattore di riempimento}$
    \item $B = \left \lceil \frac{T}{F \cdot f} \right \rceil$ viene allocato 
    il numero di bucket di puntatori uguale a $B$.
\end{itemize}
Viene definita una funzione di folding: 
\[
    \varphi : K \to \mathbb{Z}^+ \text{ trasforma chiavi in interi}
\]
La funzione di hashing è data da:
\[
    h : \mathbb{Z}^+ \to B 
\]
\subsubsection{Ricerca in un indice hash}

\begin{enumerate}
    \item Dobbiamo calcolare $b = h(\varphi(K))$ dove il costo è dato da un accesso alla memoria secondaria e quindi è zero. 
    \item Accedere al bucket $b$ dove il costo è uguale a $1$ essendo che facciamo un accesso alla pagina 
    \item Accedere alle $n$ tuple tramite puntatori dove il costo è uguale $m \le n$ dove $m$ sono il numero di pagine.

        \centering 
        \includegraphics[width=0.7\textwidth]{hash_ex.png}
        \caption{Ricerca di una chiave $K$ in un indice hash}
    \end{figure}
\end{enumerate}
